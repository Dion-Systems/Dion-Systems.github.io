Computer software began with patch cables and switches. Further advancements then allowed software to be encoded into punched cards, which could be fed into a more general-purpose computer. Not too long after, programming largely shifted away from punch cards, and towards digitally-encoded text files that a programmer could edit with a text editor program.

Progress continued. Over the next four (or so) decades, programming tools built for these text files continued to mature; we got version control systems, syntax highlighting, smarter text editor features, among many other things...

...but advancement stagnated. Over four decades after the world largely switched to storing computer programs as text files, programming largely feels the same, and in many ways, worse than it used to.

Today, code editors and development environments feel slower and dumber than they should, despite hardware having monumentally improved. They seem to fail at doing anything useful in the presence of trivial mistakes. Learning about how a program is actually working is notoriously difficult, even for experts using specialized tools like debuggers, which are specifically made for informing a user how a program is actually working. Learning how to program at all in the first place is a seemingly impossible task for many.

Why?

At Dion Systems, we have a theory: many of these problems are because of architectural decisions made a long time ago. Those decisions go largely unquestioned in the modern programming world, but we think to fix these problems, it is going to require rethinking those decisions, and starting from scratch.

Our team is focused on doing exactly this, in an effort to build a better computing world. You can read more about what we're doing to try to do this at our blog.
